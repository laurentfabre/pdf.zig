## Ranked findings

### 1. [P0] Redefine “XY-Cut++” as a **tagged/lattice/stream hybrid**, not literal page-wide XY-cut
A literal recursive XY-cut over text boxes is **not** the 2026 SOTA for born-digital PDF tables. For `pdf.zig`, the right v1.2 design is:

1. **Tagged path first**: use `/StructTreeRoot` + MCIDs when the PDF already exposes `/Table` → `/TR` → `/TH`/`/TD`. Your repo already parses structure trees, so this is the cheapest high-value win.
2. **Lattice path second**: detect ruled tables from PDF vector ops (lines/rectangles/strokes), not from raster OCR. This is the classic strength of Camelot/pdfplumber-style systems on born-digital PDFs.
3. **Stream path third**: for unruled hotel menus / spa price lists, detect table regions from repeated x-anchors, baseline regularity, whitespace gutters, font consistency, decimal/currency alignment, and recurring row bands.
4. **Canonical cell grid**: infer row/col spans, then serialize from a cell grid; do **not** make Markdown the canonical representation.
5. **Confidence + abstention**: wrong table structure is often worse for LLM grounding than emitting plain text plus `low_confidence_table=true`.

That is still “no-ML”, still Zig-implementable, and much closer to what strong practitioners actually ship for born-digital PDFs than a pure XY-cut.

- **Ship natively**: tagged/lattice/stream heuristic extractor in Zig.

### 3. [P1] For v1.2 acceptance, gate on **table recall + structure**, not Markdown exact match
For tables, the wrong metric will send you in the wrong direction.

#### Recommended metrics
1. **Table detection**: table-level **Recall / Precision / F1** at IoU ≥ 0.5, and a stricter Recall at IoU ≥ 0.8.
2. **Primary structure metric**: **GriTS\_Con** (and track GriTS\_Top / GriTS\_Loc).  
   Why: PubTables-1M/TATR made GriTS the cleanest public metric for grid correctness.
3. **Primary domain metric**: **TEDS-Struct** on normalized HTML.  
   Why: hotel tables often have merged headers / section bands; TEDS-Struct captures that without over-penalizing minor text noise.
4. **Secondary text+structure metric**: **full TEDS**.
5. **Debug metric**: **cell-level F1** on `(row, col, rowspan, colspan, text)` tuples.

#### What to gate on
- **Hard gate**: table recall @ IoU≥0.8
- **Primary quality gate**: **TEDS-Struct**
- **Public-benchmark comparability gate**: **GriTS\_Con**
- **Debug-only / dashboard**: full TEDS, GriTS\_Loc, cell F1

#### What not to use as the main gate
- **RVL-CDIP**: document classification, not table extraction.
- **Single composite OmniDocBench score**: useful for broad parsers, too coarse for a `pdf.zig` milestone.
- **Markdown string diff**: pipe-table text hides merged-cell and header-band failures.

Rule of thumb: **OmniDocBench-style systems separate structure from text**; `pdf.zig` should do the same.

### 4. [P1] Use a **hybrid corpus**; the main release gate must be **Alfred hotel tables**
Public datasets are necessary, but they are not sufficient for your domain.

#### Dataset recommendation
**Use both**:
- **Public sanity/regression set**
  - **PubTables-1M**: best public external benchmark for born-digital table structure.
  - **FinTabNet**: closer to your numeric/currency-heavy price tables than scientific papers are.
- **Domain gate**
  - Build an **Alfred hotel table gold set** from room-rate sheets, spa menus, dining hours, factsheets.

#### What to skip for v1.2
- **ICDAR 2019/2021/2023** as the primary gate: more relevant once you care about scanned/image/camera tables in v1.3 OCR land.
- **TableBank** as the primary gate: useful historically, but less aligned than PubTables-1M for born-digital structure quality.

#### Minimum viable corpus size
For a credible v1.2 gate, I would not trust less than:
- **Public subset**: ~150–250 tables total
- **Alfred gold**: **120–150 manually verified tables** across ~40–60 PDFs

If you can only afford one strong internal set, do:
- ~50 ruled tables
- ~50 unruled “stream” tables
- ~20 rotated / multi-page / merged-header edge cases

Below ~75 gold tables, TEDS will swing too much on a handful of nasty merged-header docs.

### 5. [P1] Canonical output should be a **structured table record**; Markdown is a derived view
For LLM ergonomics, the right answer is **both**, but with a clear canonical source.

#### Recommended wire model
- **Canonical internal representation**: cell grid
- **Canonical NDJSON addition**: one `kind:"table"` record per detected table
- **Backward-compatible page output**: keep inline table rendering in page markdown when possible

#### Suggested `table` record fields
- `doc_id`, `source`, `page`, `table_id`
- `bbox`
- `engine` (`native`, `docling`, `tatr`, `paddle`, `mineru`, `vlm`)
- `confidence`
- `rotation_degrees`
- `continued_from`, `continued_to`, `table_group_id`
- `n_rows`, `n_cols`
- `header_rows`, `header_cols`
- `cells[]` where each cell has:
  - `r`, `c`, `rowspan`, `colspan`
  - `bbox`
  - `text_raw`
  - `text_norm`
  - optional `is_header`
- derived serializations:
  - `html`
  - `markdown` (only if representable)
  - `csv` (only if rectangular / no spans)

#### Inline rendering policy
- **Simple rectangular tables**: inline GitHub pipe table
- **Merged cells / multi-row headers**: inline HTML table block or placeholder anchor
- Always emit the structured `table` record

#### Docling alignment
Yes, it is worth aligning **at the table-object level**, not at the whole-document-envelope level. The useful Docling ideas to mirror are roughly:
- page provenance (`prov` / bbox)
- `num_rows`, `num_cols`
- explicit cell list
- row/col spans
- captions / notes

What is **not** worth copying is Docling’s full hierarchical document JSON. `pdf.zig` is page-streaming NDJSON, not a Pythonic DOM.

### 6. [P2] Hidden risks that will bite a born-digital hotel-table extractor
1. **“CSV-origin” PDFs**: one glyph or one short token per draw op; word synthesis quality dominates table quality.
2. **Unruled stream tables**: no lines, only repeated x-anchors.
3. **Merged section headers**: “Dinner”, “Spa Rituals”, “Peak season” spanning multiple columns.
4. **Multi-page continuations**: repeated headers and carry-over rows are common in price sheets.
5. **Rotated tables**: 90° menus and landscape rate sheets.
6. **Currency / units / decimals**: `€ 1,250`, `1.250,00`, `AED`, `pp`, `min`, `sqm`, `kcal`; naive column splitting breaks badly.
7. **Scientific notation / footnotes / superscripts**: affects cell assignment and header detection.
8. **RTL / Arabic / Hebrew**: visual column order can invert; numeric cells often remain LTR.
9. **CJK**: no whitespace word boundaries; column inference must work on glyph clusters.
10. **Tables encoded as images**: native v1.2 should abstain and hand off to OCR/v1.3.
11. **Bad tagged PDFs**: structure tree exists but lies; tagged path needs sanity checks.
12. **Nested tables / sidebars**: especially in brochures/factsheets.

---

### 7. [P2] What a senior table-extraction practitioner would push for
1. **Use PDF-native signals first**. If the file already gives you text boxes, vector lines, and structure tags, do not rasterize unless you must.
2. **Separate the stages**. Detection, grid induction, and cell text assignment should be independently inspectable in debug output.
3. **Make HTML/cell-grid canonical**. Markdown/CSV are views; they are not rich enough for spans.
4. **Add confidence and abstention**. A low-confidence structured table plus plain text is often better than a wrong high-confidence table.
5. **Treat multi-page continuation as first-class**. Don’t make downstream RAG stitch tables back together.
6. **Stratify evals by table type**. A single mean score hides “ruled good / unruled bad”.
7. **Save failure artifacts**. For every failed table, keep bbox overlay + inferred grid + serialized HTML. That accelerates iteration more than aggregate scores do.

## Bottom-line recommendation

**Best v1.2 architecture**
- Native Zig, no-ML default.
- Implement **tagged → lattice → stream** table extraction.
- Keep “XY-Cut++” as the milestone label if you want, but write the design as a **hybrid PDF-native table extractor**, not a literal recursive XY-cut algorithm.

- Do **not** embed ML in the binary.
- Do **not** use MinerU or VLM APIs as the default architecture.

**Best v1.2 acceptance gates**
- Main gate: **Alfred hotel gold set**
- Metrics: **table recall @ IoU≥0.8 + TEDS-Struct + GriTS\_Con**
- Secondary dashboards: full TEDS, GriTS\_Loc, cell F1

**Best output shape**
- Keep page markdown for compatibility
- Add first-class `table` NDJSON records
- Make the cell grid / HTML the canonical representation
