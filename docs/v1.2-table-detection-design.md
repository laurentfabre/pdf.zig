# pdf.zig v1.2 — Table detection design

> **Status**: design phase, post-GA. Branch `v1.0.1-prep` then `v1.2/tables`. Synthesizes the Claude ↔ Codex deep-research loop fired on 2026-04-26 (transcript at `audit/v1.2-codex-research.md`) plus the SOTA survey.
>
> **Headline reframe**: "XY-Cut++" stays as the milestone label, but the actual algorithm is a **tagged → lattice → stream hybrid**, not a literal recursive XY-cut. PDF-native signals (structure tree, vector strokes, x-anchor regularity) outperform pure geometric XY-cut on born-digital hotel tables — which is the entire v1.2 use case.

---

## 1. Goals / non-goals

### Goals
- Detect and extract tables from born-digital PDFs without embedded ML, keeping the 647 KB static binary.
- Emit `kind:"table"` NDJSON records with a structured cell grid (rows, columns, spans, bboxes), in addition to the existing inline Markdown rendering.
- Match or exceed Docling's TableFormer on the Alfred hotel-table gold set, measured by **TEDS-Struct + GriTS_Con + table recall @ IoU≥0.8**.
- Optional sidecar: Docling (best ops) or TATR (smallest serious ML) for hard pages, *not* the default.

### Non-goals (deferred to v1.3 / v1.x)
- OCR for image-only tables (NG4 — v1.3 ocrmypdf shell-out).
- Vision-LLM extraction (Claude Opus 4.7 vision, GPT-5 vision) — non-deterministic, expensive, network-bound; useful manual fallback only.
- MinerU 2.5-Pro integration — AGPL + 100 MB+ model + GPU-oriented; misaligned with pdf.zig's binary-distribution constraint.
- Hand-drawn / camera-photo tables.

## 3. Algorithm: tagged → lattice → stream hybrid

### 3.1 Pass A — Tagged path (cheapest, highest precision)

Use the upstream `/StructTreeRoot` parser (already in `src/structtree.zig`). For PDFs with `/Table` → `/TR` → `/TH`/`/TD` MCID structure:

1. Walk the structure tree per page.
2. For each `/Table` node, collect its `/TR` rows → `/TH` / `/TD` cells.
3. For each cell, look up its MCID in the content stream → bounding box + glyph cluster.
4. Build a (row, col, rowspan, colspan, bbox, text) tuple list.
5. Emit one `kind:"table"` record per `/Table` node with `engine:"native_tagged"` and `confidence: 1.0` (modulo sanity checks below).

**Sanity checks** (tagged PDFs sometimes lie):
- Reject if any cell has zero glyphs *and* a non-empty bbox (tag exists but content lost).
- Reject if `n_rows * n_cols < n_cells * 0.5` (degenerate spans).
- Demote rejected tables to Pass B/C.

### 3.2 Pass B — Lattice path (vector-line driven)

For PDFs without usable structure tree, look for ruled tables via vector graphics ops:

1. During content-stream interpretation, collect every `re` (rectangle), `m`/`l` (move/line) operation that produces a horizontal or vertical stroke.
2. Cluster strokes by their orientation + collinearity (within 1 pt tolerance).
3. Two parallel horizontal strokes + two parallel vertical strokes that close a rectangle → table candidate.
4. Detect interior horizontal strokes (row separators) and interior vertical strokes (column separators) → cell grid.
5. Assign text to cells by intersecting glyph centers with cell bboxes.
6. Emit `kind:"table"` with `engine:"native_lattice"` and `confidence: 0.8 + bonus_for_complete_grid`.

This is the **Camelot lattice** approach, adapted to Zig and integrated with the existing content-stream interpreter (`src/interpreter.zig`).

### 3.3 Pass C — Stream path (unruled, hardest)

For unruled tables (the typical hotel menu / spa price list):

1. Group glyph clusters into "lines" by Y-coordinate (already done by `layout.zig::sortGeometric`).
2. For each candidate region (e.g. paragraph block), compute the histogram of per-line `x_anchor` positions. Strong bimodal/trimodal peaks at consistent X = column boundaries.
3. Verify recurring row bands: same Y-spacing across ≥4 lines, same column-anchor pattern.
4. Verify decimal/currency alignment: in numeric columns, decimal point or currency symbol aligns within ±2 pt across rows.
5. Reject if column count < 2 or row count < 3 (too small to be a table).
6. Emit `kind:"table"` with `engine:"native_stream"` and `confidence: 0.5 + heuristic_score`.

### 3.4 Pass D — Confidence + abstention

Per Codex's senior-practitioner finding: a wrong table structure is often worse for LLM grounding than no table. So:

- `confidence < 0.6` → emit the page's plain-markdown rendering (no `kind:"table"` record), plus a `quality_flag:"low_confidence_table"` warning on the page record.
- `0.6 ≤ confidence < 0.8` → emit `kind:"table"` AND keep the markdown row representation; downstream consumer chooses.
- `confidence ≥ 0.8` → emit `kind:"table"` and replace the table region in markdown with `[table table_id=N]` placeholder.

### 3.5 Multi-page continuation (first-class)

A `table_group_id` UUID groups tables that span page boundaries. Detection heuristic:
- Same column count, same column header text (or empty header), starts at top of page N+1 within `header_y_threshold` of where N's table ended.

`continued_from` / `continued_to` fields on each `table` record point at the previous/next page's `table_id`.

## 4. NDJSON record shape

```json
{
  "kind": "table",
  "doc_id": "019dca97-…",
  "source": "spa-menu.pdf",
  "page": 3,
  "table_id": 1,
  "table_group_id": null,
  "continued_from": null,
  "continued_to": null,
  "engine": "native_lattice",
  "confidence": 0.92,
  "rotation_degrees": 0,
  "bbox": [72.0, 540.5, 524.0, 720.0],
  "n_rows": 12,
  "n_cols": 4,
  "header_rows": 1,
  "header_cols": 0,
  "cells": [
    {"r": 0, "c": 0, "rowspan": 1, "colspan": 4, "is_header": true, "bbox": [72.0, 700.0, 524.0, 720.0], "text_raw": "DINNER", "text_norm": "Dinner"},
    {"r": 1, "c": 0, "rowspan": 1, "colspan": 1, "is_header": true, "bbox": [72.0, 680.0, 200.0, 700.0], "text_raw": "Dish", "text_norm": "Dish"},
    {"r": 1, "c": 1, "rowspan": 1, "colspan": 1, "is_header": true, "bbox": [200.0, 680.0, 320.0, 700.0], "text_raw": "Description", "text_norm": "Description"},
    "..."
  ],
  "html": "<table><tr><td colspan=\"4\"><strong>DINNER</strong></td></tr>…</table>",
  "markdown": "| Dish | Description | Calories | Price |\n|---|---|---|---|\n…",
  "csv": null
}
```

- `cells[]` is the **canonical** representation. Spans, headers, bboxes are first-class.
- `html` + `markdown` + `csv` are derived views. `csv` is `null` when the table has merged cells (CSV can't represent spans).
- Page records continue to render the table inline in markdown for backward compatibility — gated on confidence per §3.4.

## 5. Eval methodology

### 5.1 Metrics (per Codex finding #3)

| Metric | Role | Source |
|---|---|---|
| **Table recall @ IoU≥0.8** | Hard gate | Per-PDF count: detected / ground-truth |
| **TEDS-Struct** | Primary quality gate | Per-table tree-edit-distance on normalized HTML structure (no text content) |
| **GriTS_Con** | Public-benchmark comparability | Grid Table Similarity, content variant |
| Full TEDS | Dashboard | TEDS-Struct + text content matching |
| GriTS_Loc | Dashboard | Grid bbox-aware variant |
| Cell F1 | Debug | (row, col, rowspan, colspan, text) tuple precision/recall |

**Acceptance gates for v1.2 release**:
- Table recall @ IoU≥0.8 ≥ **0.90** on the Alfred hotel gold set
- TEDS-Struct ≥ **0.85** on the Alfred hotel gold set
- GriTS_Con ≥ **0.75** on PubTables-1M sanity subset (Microsoft baseline ≈ 0.985 — we are intentionally lower because we ship a heuristic, not a 29M-param model)

### 5.2 Stratified reporting

Per Codex senior-practitioner finding #6, score by table-class strata so single-mean averages don't hide per-class regressions:

- Ruled tables (lattice path)
- Unruled tables (stream path)
- Tagged tables (tagged path)
- Multi-page continuation
- Merged-header section bands (e.g. "Dinner" spanning 4 cols)
- Rotated tables

Per-stratum recall + TEDS-Struct reported in `bake-off/scorecard/v1.2-tables.tsv`.

## 6. Dataset plan

### 6.1 Hybrid corpus per Codex finding #4

**Public sanity / regression** (~150–250 tables total):
- **PubTables-1M** subset: 100 born-digital structure-rich tables. Used as the GriTS_Con regression baseline.
- **FinTabNet** subset: 50 numeric/currency-heavy tables. Closer to hotel rate sheets than PubTables-1M's scientific papers.
- **No ICDAR-2019 / TableBank** as primary gate (more relevant once v1.3 OCR ships).

**Domain gate (Alfred hotel gold set)** — see §6.2.

### 6.2 Alfred hotel-tables gold set (BUILDS THIS WEEK)

Target: **120 manually verified tables across ~50 PDFs**, distributed:

| Stratum | Count | Sourcing |
|---|---|---|
| Ruled price tables | 30 | spa price lists with horizontal/vertical rules |
| Unruled menu tables | 35 | dining/breakfast menus with stream layout |
| Unruled factsheet tables | 25 | hotel factsheets with key/value rows |
| Multi-page continuation | 15 | brochures with tables spanning ≥2 pages |
| Rotated tables | 8 | landscape-orientation rate cards |
| Tagged-PDF tables | 7 | accessibility-marked PDFs with `/StructTreeRoot` |

Sourcing pool: 720 PDFs in Alfred matching `*menu*`, `*price*`, `*rate*`, `*spa*`, `*factsheet*`. Sampling stratified by hotel for diversity.

### 6.3 Annotation format

Per-table JSON file at `audit/tables-gold/<hotel_slug>/<pdf_basename>.tables.json`:

```json
{
  "doc": "spa-menu.pdf",
  "tables": [
    {
      "id": 0,
      "page": 3,
      "stratum": "unruled_menu",
      "bbox": [72.0, 540.5, 524.0, 720.0],
      "n_rows": 12,
      "n_cols": 4,
      "header_rows": 1,
      "cells": [
        {"r": 0, "c": 0, "rowspan": 1, "colspan": 4, "is_header": true, "text": "DINNER"},
        "..."
      ]
    }
  ]
}
```

The annotation script `audit/v1_2_annotate.py` opens each PDF, displays detected candidate regions (from a debug build of pdf.zig), and lets the human accept / reject / correct cells per table.

### 6.4 Minimum viable corpus rationale

Codex's threshold: below ~75 gold tables, TEDS swings too much on a handful of nasty merged-header docs. Targeting 120 gives:
- Wilson 95% CI half-width ≈ 4.5pp on overall TEDS-Struct
- Per-stratum n=8–35 for directional reporting (per-stratum CIs are wider, treated as flags not gates)

## 7. Implementation plan

### 7.1 Module map (additions)

```
src/
├── tables.zig          NEW (~700 LOC): Pass A/B/C/D pipeline
├── lattice.zig         NEW (~200 LOC): vector-stroke clustering
├── stream_table.zig    NEW (~250 LOC): x-anchor histogram + row band detection
├── teds.zig            NEW (~150 LOC): TEDS-Struct evaluator (used by audit harness)
├── grits.zig           NEW (~100 LOC): GriTS_Con evaluator
└── stream.zig          MODIFY: emitTable + RecordKind.table
```

### 7.2 Sidecar protocol (optional)

For Docling sidecar fallback (`pdf.zig extract --table-engine docling`):
- pdf.zig spawns `docling --to json --tables-only <pdf>` (one-shot, not per-page).
- Reads docling's JSON output, normalizes its tables to our `kind:"table"` schema.
- Records `engine:"docling_v<n>"` and the docling commit/version in the record for reproducibility.
- Sidecar is gated by feature detection (`docling --version` succeeds) — graceful no-op if not installed.

For TATR sidecar (`--table-engine tatr`):
- Same pattern. Spawn `tatr-cli` (we'd ship a thin Python wrapper) once per document.
- TATR's HF model card pin frozen in `audit/sidecar-pins.tsv`.

## 8. Hidden risks (per Codex finding #6)

Documented for future reference; each becomes a fixture in the Alfred gold set:

1. **CSV-origin PDFs**: one glyph per draw op; word synthesis quality dominates table quality.
2. **Unruled stream tables**: no lines; only repeated x-anchors.
3. **Merged section headers**: "Dinner", "Spa Rituals", "Peak season" spanning multiple columns.
4. **Multi-page continuations**: repeated headers + carry-over rows.
5. **Rotated tables**: 90° menus and landscape rate sheets.
6. **Currency / units / decimals**: `€ 1,250`, `1.250,00`, `AED`, `pp`, `min`, `sqm`, `kcal`.
7. **Scientific notation / footnotes / superscripts**: cell assignment + header detection.
8. **RTL / Arabic / Hebrew**: visual column order can invert; numeric cells stay LTR.
9. **CJK**: no whitespace word boundaries; column inference works on glyph clusters.
10. **Image-encoded tables**: native v1.2 must abstain → `quality_flag:"image_table"` → v1.3 OCR.
11. **Bad tagged PDFs**: tagged path needs sanity checks (§3.1).
12. **Nested tables / sidebars**: especially in brochures/factsheets.

## 10. Open questions for v1.2 cycle-1 review

1. Is the tagged → lattice → stream order right, or should lattice come before tagged on PDFs where the structure tree exists but is known-bad (e.g. "tagged but lying")?
2. What's the right confidence calibration on Pass C (stream path)? The 0.5 + heuristic_score floor is a guess; calibrate against gold set.
3. Should the dataset annotation format align with PubTables-1M's XML format (for public-corpus comparability) or stay JSON-only (cleaner for our audit scripts)?
4. Sidecar invocation: per-document or per-page? Per-document is faster (1 Python startup) but pdf.zig's per-page flush ergonomics break.
5. For multi-page continuation: emit a separate `kind:"table_group"` final record with the merged grid, or leave the consumer to stitch via `continued_from` / `continued_to`?

Codex cycle-2 review on this design will fire after the gold set is built and Pass A is implemented.
